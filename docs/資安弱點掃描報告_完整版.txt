## 四、已修復問題明細

### 4.1 VULN-001: SQL 注入漏洞 [已修復]

#### 基本資訊

| 項目 | 內容 |
|------|------|
| 編號 | VULN-001 |
| 原等級 | Critical |
| 狀態 | 已修復 |
| 位置 | `services/api/src/common/middleware/tenant.middleware.ts` |

#### 問題描述

原始碼中直接將使用者控制的 schema 名稱拼接到 SQL 查詢中，未進行任何驗證或轉義。攻擊者可透過操控 API Key 來注入惡意 SQL 指令。

#### 問題程式碼（修復前）

```typescript
// 原始 vulnerable code
await this.dataSource.query(
  `SET search_path TO ${tenant.schema_name}, public`,
);
```

#### 風險分析

| 項目 | 說明 |
|------|------|
| 攻擊向量 | 透過 API Key 的 schema_name 欄位 |
| 影響範圍 | 所有使用多租戶功能的 API 請求 |
| 資料敏感性 | 高 - 可能存取其他租戶資料 |
| 可利用性 | 中 - 需要有效 API Key |

#### 修復方式

```typescript
// 白名單驗證
const VALID_SCHEMA_PATTERN = /^[a-z][a-z0-9_]*$/;

private validateSchemaName(schemaName: string): boolean {
  return VALID_SCHEMA_PATTERN.test(schemaName) && schemaName.length <= 64;
}

private escapeIdentifier(identifier: string): string {
  return '"' + identifier.replace(/"/g, '""') + '"';
}

// 使用安全的識別符
const safeSchema = this.escapeIdentifier(schemaName);
await this.dataSource.query(
  `SET search_path TO ${safeSchema}, public`,
);
```

#### 驗證方法

1. 使用常見 SQL 注入 Payload 測試：
   - `'; DROP TABLE users; --`
   - `' UNION SELECT * FROM other_schema.users;--`
2. 確認系統正確拒絕不符合规范的 schema 名稱

### 4.2 VULN-002: API 金鑰洩漏 [已修復]

#### 基本資訊

| 項目 | 內容 |
|------|------|
| 編號 | VULN-002 |
| 原等級 | High |
| 狀態 | 已修復 |
| 位置 | `.env` |

#### 問題描述

環境變數檔案中包含真實的 OpenAI API 金鑰，且該金鑰可能被提交至版本控制系統。

#### 問題程式碼（修復前）

```env
OPENAI_API_KEY=sk-proj-REDACTED
```

#### 風險分析

| 項目 | 說明 |
|------|------|
| 攻擊向量 | Git 歷史記錄、意外分享 |
| 影響範圍 | 該 API Key 關聯的 OpenAI 帳戶 |
| 財務影響 | 可能產生額外 API 費用 |
| 帳戶安全 | API Key 可能被濫用 |

#### 修復方式

1. 立即輪換暴露的 API Key
2. 更新 `.env` 檔案：
```env
OPENAI_API_KEY=sk-YOUR_OPENAI_KEY_HERE
```
3. 建立安全的 `.env.example` 範本
4. 將 `.env` 加入 `.gitignore`

### 4.3 VULN-003: JWT 密鑰弱化 [已修復]

#### 基本資訊

| 項目 | 內容 |
|------|------|
| 編號 | VULN-003 |
| 原等級 | High |
| 狀態 | 已修復 |
| 位置 | `services/api/src/auth/auth.module.ts` |

#### 問題描述

JWT_SECRET 使用預設值或弱密鑰，攻擊者可能透過猜測或取得密鑰來偽造 JWT Token。

#### 問題程式碼（修復前）

```env
JWT_SECRET=your-super-secret-jwt-key-change-in-production
```

#### 風險分析

| 項目 | 說明 |
|------|------|
| 攻擊向量 | 密鑰猜測、社會工程 |
| 影響範圍 | 所有 JWT 認證的請求 |
| 權限提升 | 可偽造任意用戶身份 |

#### 修復方式

```typescript
const WEAK_SECRETS = [
  'default-secret',
  'change-me',
  'your-super-secret',
  'dev-secret',
];

useFactory: (configService: ConfigService) => {
  const secret = configService.get('JWT_SECRET');
  const isWeakSecret = WEAK_SECRETS.some(weak => 
    secret.toLowerCase().includes(weak.toLowerCase())
  );
  
  if (isWeakSecret || secret.length < 32) {
    console.warn('⚠️  JWT_SECRET is too weak!');
  }
  
  return { secret, signOptions: { expiresIn: '7d' } };
}
```

#### 建議的強密鑰產生方式

```bash
# 使用 openssl 產生隨機密鑰
openssl rand -base64 32
```

### 4.4 VULN-004: CORS 設定過寬 [已修復]

#### 基本資訊

| 項目 | 內容 |
|------|------|
| 編號 | VULN-004 |
| 原等級 | Medium |
| 狀態 | 已修復 |
| 位置 | `services/api/src/main.ts` |

#### 問題描述

CORS 設定允許所有來源 (`origin: '*'`)，增加跨站請求偽造（CSRF）攻擊面。

#### 問題程式碼（修復前）

```typescript
cors: {
  origin: '*',  // 允許所有來源
  methods: 'GET,HEAD,PUT,PATCH,POST,DELETE',
  credentials: true,
}
```

#### 風險分析

| 項目 | 說明 |
|------|------|
| 攻擊向量 | 恶意網站發起跨域請求 |
| 影響範圍 | 所有 API 端點 |
| 風險類別 | CSRF、信息洩漏 |

#### 修復方式

```typescript
const getCorsOrigins = (): string[] => {
  const origins = process.env.CORS_ORIGINS;
  if (!origins || origins === '*') {
    return [
      'http://localhost:3000',
      'http://localhost:3001',
      'http://localhost:5173',
      'http://localhost:5174',
      'http://localhost:8080',
    ];
  }
  return origins.split(',').map(o => o.trim());
};

const app = await NestFactory.create(AppModule, {
  cors: {
    origin: getCorsOrigins(),
    methods: 'GET,HEAD,PUT,PATCH,POST,DELETE',
    credentials: true,
    allowedHeaders: ['Content-Type', 'Authorization', 'x-api-key'],
  },
});
```

### 4.5 VULN-005: 缺少 Rate Limiting [已修復]

#### 基本資訊

| 項目 | 內容 |
|------|------|
| 編號 | VULN-005 |
| 原等級 | Medium |
| 狀態 | 已修復 |
| 位置 | `services/api/src/app.module.ts` |

#### 問題描述

系統未實作 API 請求頻率限制，容易遭受暴力破解、服務阻斷攻擊（DoS）。

#### 風險分析

| 攻擊類型 | 說明 |
|----------|------|
| 暴力破解 | 大量嘗試登入 credential |
| DoS 攻擊 | 耗盡伺服器資源 |
| API 濫用 | 超出合理使用範圍 |

#### 修復方式

1. 安裝 `@nestjs/throttler` 套件
2. 設定三層級限制：

```typescript
import { ThrottlerModule } from '@nestjs/throttler';

ThrottlerModule.forRoot([
  {
    name: 'short',
    ttl: 1000,   // 1 秒
    limit: 10,   // 每秒最多 10 個請求
  },
  {
    name: 'medium',
    ttl: 10000,  // 10 秒
    limit: 50,   // 每 10 秒最多 50 個請求
  },
  {
    name: 'long',
    ttl: 60000,  // 1 分鐘
    limit: 200,  // 每分鐘最多 200 個請求
  },
])
```

### 4.6 VULN-006: 預設密碼殘留 [已修復]

#### 基本資訊

| 項目 | 內容 |
|------|------|
| 編號 | VULN-006 |
| 原等級 | Low |
| 狀態 | 已修復 |
| 位置 | `services/admin-dashboard/src/pages/Login.tsx` |

#### 問題描述

登入表單包含預設的 email 與密碼，可能在正式環境中被利用。

#### 問題程式碼（修復前）

```typescript
const [email, setEmail] = useState('admin@demo.com')
const [password, setPassword] = useState('admin123')
```

#### 修復方式

```typescript
const [email, setEmail] = useState('')
const [password, setPassword] = useState('')
```
